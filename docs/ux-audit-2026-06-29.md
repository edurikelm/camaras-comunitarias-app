## CRÍTICO (bloquea pivote a producto)

### C1. `/api/communities/[communityId]/invitations` devuelve 500 con mensaje genérico
- **Reproducción**: Como ADMIN de comunidad, ir a `/dashboard` → click "Crear invitación" → click "Generar codigo" → toast muestra `Internal server error`.
- **Network**: `POST /api/communities/51ffda0d-.../invitations` → 500 con body `{"error":"Internal server error"}`.
- **Impacto**: Función core del producto (incorporar vecinos) totalmente rota para el único caso real probado. No hay manera de incorporar nuevos miembros.
- **Root cause probable**: El error real nunca llega al usuario; `mapDomainErrorToResponse` lo silencia con `console.error` que no aparece en `dev.log`. El servidor de Next.js 16 con Turbopack tampoco lo captura de forma inspeccionable.
- **Acción**: Reproducir en local con logging mejorado, identificar el `error instanceof DomainError` que se está escapando. Probable falta de `actor.communityId` o membership lookup fallando.
- **Severidad funcional**: ALTA (bloqueante). **Severidad UX**: igual de bloqueante — el admin nunca sabe qué pasó.

### C2. `Login` siempre redirige a `/platform` hardcodeado, ignorando rol
- **Código**: `src/app/login/page.tsx:48` — `router.push("/platform")` sin chequear rol.
- **Reproducción**: Vecina NEIGHBOR hace login → aterriza en `/platform` → ve `NoPermissionState` ("No tienes permisos de administrador de plataforma").
- **Esperado**: Login debería ir al dispatcher `/` (`src/app/page.tsx`) que ya tiene la lógica correcta (PLATFORM_ADMIN → /platform, ACTIVE member → /dashboard, PENDING/none → /request-membership).
- **Fix de 1 línea**: cambiar `router.push("/platform")` por `router.push("/")`.
- **Severidad**: ALTA. Cualquier vecino que intente loguearse choca con este bug.

### C3. Registro no captura `name` antes de crear el usuario en Supabase Auth
- **Código**: `src/app/register/page.tsx:40-44` — `supabase.auth.signUp({ email, password })` se llama ANTES del POST a `/api/register`.
- **Impacto**: Si el usuario se equivoca al tipear el email (un typo en una pass de un solo intento), el usuario queda creado en Supabase Auth huérfano. Si el POST a `/api/register` falla, el usuario tiene que volver a registrarse con un email nuevo (porque Supabase puede haber quedado con el user pero sin profile local → loop).
- **Fix**: Validar `name` + email local antes de signUp, o usar el `data` de signUp directamente para crear el User local sin segundo round-trip.
- **Severidad funcional**: MEDIA. **Severidad UX**: ALTA (genera confusión y usuarios duplicados).

## ALTA (afecta confianza y adoption)

### A1. Errores crudos de Supabase Auth se filtran al usuario
- **Reproducción**: `/register` con email `vecina.test.001@example.com` → muestra literal `Email address "vecina.test.001@example.com" is invalid`.
- **Reproducción**: `/login` con usuario recién registrado (sin confirmar email) → muestra literal `Email not confirmed`.
- **Impacto**: Mensajes técnicos en inglés, sin tono del producto, exponen detalles internos (formato esperado de emails). En comunidad civil suena a bug, no a producto profesional.
- **Fix**: Mapear códigos de error de Supabase a mensajes en español del dominio:
  - `email_address_invalid` → "El formato del correo no es válido. Verificá e intentá de nuevo."
  - `email_not_confirmed` → "Revisá tu casilla y confirmá el correo para activar la cuenta."
  - `invalid_credentials` → "Correo o contraseña incorrectos."
  - `over_email_send_rate_limit` → "Demasiados intentos. Esperá un momento."
- **Severidad**: ALTA.

### A2. Crear comunidad pide "Auth Provider ID (UUID)" — fricción enorme para el PLATFORM_ADMIN
- **Reproducción**: `/platform` → "Crear comunidad" → diálogo pide `authProviderId` (UUID de Supabase Auth) + email + nombre del primer admin.
- **Impacto**: Un admin real no tiene idea de dónde sacar ese UUID. Flujo real esperado:
  1. Admin futuro se registra en `/register` (queda en Supabase Auth con un UUID que ni él ni el PLATFORM_ADMIN ven).
  2. PLATFORM_ADMIN abre Supabase Dashboard → Authentication → Users → copia UUID.
  3. PLATFORM_ADMIN vuelve a `/platform` → pega UUID.
- **Fix correcto**: Cambiar el flujo a **generar invitación por email**. El PLATFORM_ADMIN tipea email del futuro admin, el sistema crea una invitación con token, el futuro admin hace click en un magic link → queda registrado Y asignado como ADMIN de esa comunidad. El UUID de Supabase se crea al confirmarse el link, no antes.
- **Severidad funcional**: MEDIA (hay workaround). **Severidad UX**: ALTA (3 pasos manuales con un valor técnico opaco).

### A3. No hay auto-login post-registro
- **Código**: `src/app/register/page.tsx:80-83` — `setTimeout(() => router.push("/login"), 2000)`.
- **Impacto**: Después de "Cuenta creada correctamente" el usuario tiene que volver a tipear email + password. Si se equivocó al confirmar password, ni se entera hasta login.
- **Fix**: Si `signUpData.session` viene no-null (auto-confirm habilitado), usar la sesión directamente. Si no, mostrar pantalla de "Revisá tu correo para confirmar" en vez de redirigir al login ciegamente.
- **Severidad**: ALTA.

### A4. Botones "Próximamente: …" disabled en toda la app
- **Páginas**: `/dashboard` ("Crear sector", "Registrar camara"), `/cameras` ("Próximamente: registrar camara", "Registrar primera camara"), `/incidents` ("Reportar incidente" duplicado como action y como empty state).
- **Impacto**: La UI muestra opciones que no funcionan. El usuario hace click → nada. Confunde sobre el estado del producto. Diseño admite reasumir que estas features existen pero están rotas.
- **Recomendación**: Quitar los botones hasta que la feature esté lista. Si querés indicar roadmap, usar un badge "Próximamente" sin botón clickeable (shadcn ya tiene `Badge`). El `<Empty>` con `EmptyContent` queda OK para describir pero sin botón fantasma.
- **Severidad**: ALTA (engaña al usuario sobre el producto).

### A5. Empty state + "Próximamente" contradictorio en `/incidents`
- **Página**: `/incidents` cuando no hay incidentes.
- **Render**: Empty state dice "No hay incidentes reportados" con botón "Próximamente: Reportar incidente" disabled. Pero el header tiene un botón "Crear incidente" ARRIBA a la derecha que SÍ funciona y abre el diálogo.
- **Impacto**: El usuario ve dos CTAs para la misma acción, una rota y una viva. Inconsistencia visual.
- **Fix**: Empty state debe usar el mismo trigger que el header (mismo componente `CreateIncidentDialog`), o quitar el botón del empty state.
- **Severidad**: MEDIA.

### A6. Nav muestra TODAS las secciones a TODOS los roles
- **Código**: `src/components/domain/route-shell.tsx:10-16` — array fijo `navigationItems`.
- **Reproducción**: Como NEIGHBOR (no admin) hago login → veo link "Plataforma" en el nav → click → `NoPermissionState` rojo.
- **Impacto**: Revela secciones que el usuario no puede usar. Confunde y degrada confianza. Un PLATFORM_ADMIN no es miembro "típico" pero igual ve "Camaras" e "Incidentes" — eso está bien porque puede tener membresía; pero un NEIGHBOR puro no debería ver "Plataforma".
- **Fix**: Pasar `viewerRole` al `RouteShell` y filtrar items según rol. Para MVP: ocultar "Plataforma" si no es PLATFORM_ADMIN.
- **Severidad**: MEDIA-ALTA.

## MEDIA (afecta polish y eficiencia)

### M1. Contraste WCAG: badges destructive y Alert destructive fallan AA
- **Medición real** (calculada contra `--destructive: #ef4444` y white):
  - `text-white on bg-destructive` = **3.60:1** — falla AA para texto normal (necesita 4.5:1).
- **Uso actual**:
  - `<Badge variant="destructive">` (member approval "Rechazar", incidents `severity=CRITICAL`).
  - `<Alert variant="destructive">` (`NoPermissionState`, "SOS no cambia permisos de camara" en `/incidents`).
- **Impacto**: Lectura cansada para baja visión, falla a11y AA que exige la sección Accessibility de DESIGN.md ("Mantener contraste suficiente").
- **Fix**: Cambiar `--destructive` a un rojo más oscuro, p.ej. `hsl(0 73% 45%)` ≈ `#cf2e2e`, que da ~5.5:1 con blanco. O usar `text-destructive-foreground` con un fondo más oscuro dedicado para el caso Alert.
- **Severidad**: MEDIA (rompe a11y WCAG AA, no es bloqueante legal pero sí de polish).

### M2. `RouteShell` no maneja el caso "PENDING membership" en `/cameras` e `/incidents`
- **Páginas**: `/cameras/page.tsx` y `/incidents/page.tsx` no tienen el bloque "Membresía pendiente" que sí tiene `/dashboard`.
- **Reproducción**: PENDING member entra a `/incidents` directo → ve `NoPermissionState` "Membresía requerida" (no es la misma semántica: SÍ tiene membresía, solo está pendiente).
- **Impacto**: Mensaje equivocado. Un PENDING que es miembro pero no activado debería ver "Membresía pendiente de aprobación", no "Membresía requerida".
- **Fix**: Refactor del helper `getMembership()` (ya existe en `/incidents/page.tsx:47`) para distinguir 4 estados: no-session, no-db-user, no-membership, PENDING, BLOCKED, ACTIVE. Aplicar mensajes coherentes en las 3 rutas autenticadas.
- **Severidad**: MEDIA (consistencia + info correcta).

### M3. `/login` no muestra feedback de "campo inválido" cuando email es vacío y se hace submit
- **Reproducción**: `/login` → click "Iniciar sesión" sin tipear nada → el navegador muestra "Please fill out this field" (popup nativo), no hay feedback de la app.
- **Fix**: Aceptable que el browser nativo lo haga, pero para consistencia se debería desactivar `required` y validar manualmente con Zod para tener mensajes uniformes en español.
- **Severidad**: BAJA-MEDIA.

### M4. Sin "skip link" para navegación por teclado
- **Reproducción**: Presionar Tab repetidamente desde `/login` o `/dashboard`. No hay forma de saltar al contenido principal.
- **Impacto**: Falla WCAG 2.4.1 (Bypass Blocks, Level A).
- **Fix**: Agregar `<a href="#main-content" className="sr-only focus:not-sr-only ...">Saltar al contenido principal</a>` en `RootLayout`.
- **Severidad**: MEDIA (AA fail).

### M5. Diálogos Radix no anuncian errores de validación hasta que el usuario interactúa
- **Reproducción**: Abrir "Crear incidente" → dejar `type` vacío (placeholder "Selecciona un tipo") → click "Reportar incidente".
- **Impacto**: Si la validación Zod dispara, el `FormMessage` aparece DEBAJO del campo. Para screen readers con VoiceOver/NVDA, el mensaje se anuncia cuando el usuario tab vuelve al campo, no al momento del submit. Falla WCAG 3.3.1 (Error Identification).
- **Fix**: Agregar `aria-live="assertive"` al contenedor de mensajes, o mover foco al primer campo con error tras submit fallido (Vercel composition patterns).
- **Severidad**: MEDIA.

### M6. `Live view` placeholder muestra "Camara no disponible" sin explicar el porqué
- **Página**: `/cameras`.
- **Render**: Skeleton de video + badge "Camara no disponible" + 3 badges de estado (PENDING_REVIEW, ACTIVE, PRIVATE) sin contexto.
- **Impacto**: El usuario no sabe si es bug, si su cámara está offline, o si el admin aún no la aprobó. Falta texto que diga "Tu cámara aún está en revisión por un administrador" o equivalente.
- **Fix**: Cuando `userCameras` está vacío o todas en PENDING_REVIEW, mostrar empty state contextual con copy del paso siguiente ("Esperá revisión del administrador").
- **Severidad**: MEDIA.

### M7. Mensaje de éxito en `/register` se desvanece con timeout fijo de 2s
- **Código**: `src/app/register/page.tsx:80-83` — `setTimeout(() => router.push("/login"), 2000)`.
- **Impacto**: 2 segundos es poco para usuarios con lector de pantalla. WCAG 2.2.1 (Timing Adjustable) requiere que el usuario pueda extender, pausar o desactivar el límite. Aceptable porque la duración es < 20s pero debería ser ajustable.
- **Fix**: Mostrar el mensaje de éxito sin auto-redirect, agregar botón "Ir a iniciar sesión" explícito.
- **Severidad**: BAJA.

## BAJA (nice to have, polish)

### L1. Sin favicon ni meta og:image
- **Verificación**: Lighthouse SEO 100/100 (Next.js genera defaults), pero no hay favicon custom ni og:image para compartir en redes/comunidad.
- **Severidad**: BAJA.

### L2. Sin `prefers-reduced-motion` global
- **Reproducción**: Habilitar "Reduce motion" en OS → las animaciones del Alert (slide-in/out del dialog Radix) siguen ejecutándose.
- **Fix**: Agregar `@media (prefers-reduced-motion: reduce) { animation-duration: 0.01ms !important }` en `globals.css`.
- **Severidad**: BAJA (cumple WCAG AAA opcional).

### L3. `placeholder` en algunos inputs usan ejemplos en español de Argentina, no neutro
- **Ejemplos**: "tu@correo.com", "admin@comunidad.cl" (mezcla .cl en algunos lugares).
- **Impacto**: Si bien Argentina está en target, la app referencia `.cl` en el placeholder de "Crear comunidad". Inconsistencia de locale.
- **Severidad**: BAJA.

### L4. Iconos de lucide-react sin `aria-hidden` consistente
- **Verificación**: Algunos iconos tienen `data-icon="inline-start"`, otros no. Para iconos puramente decorativos, deberían tener `aria-hidden="true"` explícito.
- **Severidad**: BAJA (Lighthouse pasa pero pulido).

### L5. `Crear comunidad` no tiene Cancelar al lado de Crear (asimetría con otros diálogos)
- **Comparación**: "Crear incidente" tiene `Cancelar` + `Reportar incidente` (footer), pero "Crear comunidad" solo tiene `Cancelar` + `Crear comunidad` en el mismo footer. En realidad SÍ están simétricos. Falsa alarma. Borrar.

### L6. `Alert` "Sin alertas críticas activas" en `/dashboard` es confuso
- **Render**: Aparece SIEMPRE en estado normal, incluso cuando no hay nada que alertar.
- **Impacto**: Ruido visual permanente. Mejor: solo mostrar si hay alertas, o cambiar el copy a "Bandeja de alertas vacía — todas las alertas internas aparecerán aquí cuando ocurran".
- **Severidad**: BAJA.

### L7. Footer con cards "Próximo paso operativo" es meta-contenido, no usuario
- **Página**: `/dashboard` al final.
- **Texto**: "Implementar servicios de dominio antes de conectar datos reales. Validar comunidad, rol, estado ACTIVE y estado de comunidad en cada consulta sensible."
- **Impacto**: Es una TODO list del dev filtrada al usuario final. Quitar o mover a `README.md`.
- **Severidad**: BAJA-MEDIA (reputación: producto se ve inacabado).
